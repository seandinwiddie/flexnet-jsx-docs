Events System

The FlexNet JSX events system is the bridge between user interactions—which are inherently impure—and the pure functional core of an application. It provides a structured, composable way to handle DOM events, manage application-wide communication, and trigger state changes without side effects, following the principles outlined in the Architecture Overview.

Event Handling Principles

The FlexNet JSX events system is built on the following core principles:

Core Principles

  • **Pure Handlers:** Event handlers should be pure functions that translate event data into actions or messages.
  • **Unidirectional Data Flow:** Events trigger state changes through a central store, which then causes the UI to re-render. As described in the architecture guide, data flows in one direction.
  • **Isolate Side Effects:** All side effects (like `console.log` or DOM manipulation) must be handled by the Effect system, not directly in event handlers.
  • **Functional Composition:** Complex event logic is built by composing smaller, pure functions using `compose` and `pipe`.
  • **Decoupled Communication:** The `EventBus` system should be used for cross-component communication instead of direct prop-drilling.

DOM Event Handling and State Integration

Events drive state changes. The correct pattern is for an event handler to dispatch a descriptive action to the central store. A pure reducer function then computes the next state, and the UI re-renders.

import { createElement, render } from '../../systems/render/functions.js';
import { createStore } from '../../systems/state/functions.js';

// 1. Define a pure reducer function.
const reducer = (state, action) => {
    switch (action.type) {
        case 'INCREMENT':
            return { ...state, count: state.count + 1 };
        case 'DECREMENT':
            return { ...state, count: state.count - 1 };
        default:
            return state;
    }
};

// 2. Create an immutable store.
const store = createStore(reducer, { count: 0 });

// 3. The component dispatches actions onClick. It does not know the implementation.
const Counter = ({ count, dispatch }) =>
  createElement('div', null,
    createElement('h1', null, `Count: ${count}`),
    createElement('button', { onClick: () => dispatch({ type: 'DECREMENT' }) }, '-'),
    createElement('button', { onClick: () => dispatch({ type: 'INCREMENT' }) }, '+')
  );


// 4. The main render function subscribes to the store.
const renderApp = () => {
    const currentState = store.getState();
    const vnode = Counter({
        count: currentState.count,
        dispatch: store.dispatch // Pass the dispatch function to the component.
    });
    // The render function is defined in /src/systems/render/functions.js
    render(vnode, document.getElementById('root'));
};

store.subscribe(renderApp);

// Initial render
renderApp();

Application-Wide Communication with `createEventBus`

For communication between decoupled parts of an application, FlexNet JSX provides the `createEventBus` system. This avoids complex prop-drilling and allows different systems to react to named events.

Signature

// From: /src/systems/events/functions.js
// As defined in the API Reference.

const createEventBus = () => {
    const subscribers = new Map();
    
    return {
        emit: (event, data) =>
            Maybe.fromNullable(subscribers.get(event))
                .map(subs => subs.forEach(fn => fn(data))),
        on: (event, callback) => {
            const subs = subscribers.get(event) || new Set();
            subs.add(callback);
            subscribers.set(event, subs);
            return () => subs.delete(callback);
        }
    };
};

Example: User Notifications

Imagine a scenario where a successful API call in one part of the app needs to trigger a notification toast in another. The `EventBus` is perfect for this.

import { createEventBus } from '/src/systems/events/functions.js';
import { Maybe } from '/src/core/types/maybe.js';
import { createElement } from '../../systems/render/functions.js';

// 1. Create a single, shared event bus for the application.
const appEventBus = createEventBus();

// 2. A notification component listens for 'show-notification' events.
// This example uses pseudo-code for state and effects (e.g., local state `message`
// and lifecycle hooks) to illustrate the concept. In a real app, you would
// use the framework's state and effect management systems.
const NotificationManager = () => {
    let message = Maybe.Nothing();
    
    const showMessage = (data) => {
        message = Maybe.Just(data.message);
        // In a real app, this would trigger a re-render via the state system.
    };

    const unsubscribe = appEventBus.on('show-notification', showMessage);

    // The returned unsubscribe function should be called on cleanup to prevent memory leaks.
    // e.g., cleanupEffect(unsubscribe);

    return message.map(msg =>
        createElement('div', { className: 'toast' }, msg)
    ).getOrElse(null);
};

// 3. Another part of the application emits an event.
// The `response` is an Either or Result type, as described in the Error Handling guide.
const handleApiResponse = (response) => {
    // The `emit` function is safe, as it returns a `Maybe` which handles cases
    // where no listeners are subscribed to the event.
    response.fold(
      (error) => appEventBus.emit('show-notification', { message: `Error: ${error}` }),
      (data) => appEventBus.emit('show-notification', { message: 'Data saved successfully!' })
    );
};

Best Practices

  • Dispatch Actions from Handlers: DOM event handlers should only be responsible for dispatching actions to the state store.
  • **Keep Reducers Pure:** All state transformation logic must happen inside pure reducer functions.
  • **Use EventBus for Cross-Cutting Concerns:** Use the `EventBus` for notifications, logging, or any communication between loosely coupled components.
  • **Compose Functional Pipelines:** For complex input handling, use `compose` to create a pipeline that extracts, validates, and transforms data before creating an action. See the Security Guide for validation patterns.
  • **Manage Subscriptions:** Always use the returned `unsubscribe` function from `eventBus.on()` to clean up subscriptions when a component unmounts to prevent memory leaks.
