Security Best Practices

FlexNet JSX is designed with security as a core principle. Its architecture promotes practices that help mitigate common web vulnerabilities. This guide outlines the built-in security features and best practices for developing secure applications.

Content Escaping and XSS Prevention

Cross-Site Scripting (XSS) is one of the most common vulnerabilities in web applications. FlexNet protects against this by default.

Automatic Escaping

Any dynamic data you render in your components is automatically converted to a string and escaped before being inserted into the DOM. This means that if a user tries to inject a <script> tag or other malicious HTML into an input field, it will be rendered as harmless text rather than being executed. 

// User input with a malicious script
const userInput = `<img src="invalid" onerror="alert('XSS Attack!')">`;

// The component renders the input
const UserComment = ({ text }) => <div>{text}</div>;

// The output in the DOM will be:
// <div>&lt;img src="invalid" onerror="alert('XSS Attack!')"&gt;</div>
// The script will NOT execute.

Rendering Raw HTML

In rare cases, you might need to render HTML that you trust. For this, FlexNet provides the dangerouslySetInnerHTML prop. As its name implies, this feature should be used with extreme caution.

Warning: Improper use of dangerouslySetInnerHTML can expose your users to XSS attacks. Only use it with content that has been sanitized. 

import DOMPurify from 'dompurify';

const untrustedHtml = `<h3>User Content</h3><script>doEvil()</script>`;
const sanitizedHtml = DOMPurify.sanitize(untrustedHtml);

const SafeComponent = () => (
    <div dangerouslySetInnerHTML={{ __html: sanitizedHtml }} />
);
// The <script> tag will be removed by DOMPurify.

Immutable State

All state in a FlexNet application is immutable. Reducers must be pure functions that return a new state object instead of modifying the existing one. This has several security benefits:

  • Prevents Unintended Side Effects: It eliminates a class of bugs where different parts of the application could unintentionally modify a shared piece of state, potentially leading to security flaws.
  • Predictable State Transitions: Since state changes are explicit and controlled, it's easier to audit the flow of data and ensure that sensitive information is not being altered or exposed incorrectly.

Security Checklist

  • Do rely on FlexNet's automatic content escaping for all dynamic data.
  • Do keep your reducers pure and your state immutable.
  • Do Not use dangerouslySetInnerHTML unless you are certain the source is trusted and sanitized.
  • Do Not construct component props from user input without validation.
  • Do Not expose sensitive information in the store state that could be logged or serialized.
