Architecture Overview

FlexNet JSX is architected as a collection of modular, interoperable systems built on a purely functional core. This design emphasizes predictability, testability, and a clear separation of concerns.

High-level view of the framework's systems.

The Core

At the center of the framework is a set of core functional principles and utilities.

  • Pure Functions: Utilities like pipe, compose, and curry for building logic.
  • Functional Types: The Maybe, Either, and Result types for explicit handling of uncertainty and errors.
  • Immutability: All systems are designed to work with immutable data structures.

The Systems

The core principles are used to build several distinct but interconnected systems, each with a specific responsibility.

1. State System

This system provides the tools for managing application state, following the Redux pattern.

  • createStore: Creates the state container.
  • dispatch: Sends actions to update the state.
  • getState: Retrieves the current state.
  • subscribe: Listens for state changes.

2. Render System

This system is responsible for rendering the UI and keeping it in sync with the application state.

  • createElement: Creates virtual DOM nodes that describe the UI.
  • render: Takes a virtual DOM tree and applies it to the actual DOM using an efficient diffing algorithm.

3. Effect System

This system handles side effects, such as asynchronous operations. It acts as middleware for the state system.

  • Effects are functions that listen for specific actions.
  • They can perform tasks like fetching API data.
  • They dispatch new actions with the results of their tasks.

4. Event System

This system attaches and manages DOM event listeners (e.g., onClick, onChange) in a declarative way as part of the rendering process.

Core Operating Schema

These systems work together in a predictable cycle:

  1. The Render System creates the initial UI based on the initial state from the State System.
  2. A user interacts with the UI, which triggers an event handler from the Event System.
  3. The event handler dispatches an action.
  4. If the action is for a side effect, the Effect System handles it and may dispatch another action later.
  5. The action is received by the State System's store, which passes it to a pure reducer function.
  6. The reducer calculates the new, immutable state.
  7. The store notifies all subscribers (typically the UI's top-level render function) of the change.
  8. The Render System is called again with the new state, and it efficiently updates the DOM to match. The cycle repeats.
