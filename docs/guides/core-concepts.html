Core Concepts

Understanding the fundamental principles behind FlexNet JSX.

Functional-First Approach

FlexNet JSX is built on a foundation of functional programming principles. This means that we prioritize:

Pure Functions

Functions that always return the same output for a given input and have no side effects.

Immutability

Data that cannot be changed after creation, leading to more predictable code.

Function Composition

Building complex behavior by combining simpler functions together.

First-Class Functions

Functions that can be passed as arguments, returned from other functions, and assigned to variables.

Let's see an example of these principles in action:

import { compose, pipe } from '../../core/functions/composition.js';

// Pure function - always returns the same output for the same input
const add = (a, b) => a + b;

// Function composition - combining simple functions
const double = x => x * 2;
const increment = x => x + 1;
const square = x => x * x;

const processWithCompose = compose(square, double, increment); // Executes right-to-left
const processWithPipe = pipe(increment, double, square); // Executes left-to-right

// Immutability - creating new data instead of modifying it
const user = { name: 'Alice', age: 30 };
const updatedUser = { ...user, age: 31 }; // A new object is created
// `user` remains unchanged: { name: 'Alice', age: 30 }

JSX Runtime

The JSX runtime in FlexNet transforms JSX syntax into pure function calls that build a virtual DOMâ€”a lightweight representation of the UI. This virtual tree is then efficiently rendered and updated in the browser.

Key JSX Runtime Concepts

  • createElement: The function that JSX transpiles to, creating virtual DOM nodes.
  • Render Pipeline: A composed function that handles creating, reconciling, and patching the DOM.
  • Components: Pure functions that accept props and return virtual DOM elements.

Type System

FlexNet's type system provides functional data types to make code more predictable and robust. These types help manage common challenges like null values, errors, and asynchronous results in a safe, composable way.

M

Maybe

For handling optional values safely and avoiding null checks.

import { Maybe } from '../../core/types/maybe.js';

const getUserName = user => 
  Maybe.fromNullable(user)
    .map(u => u.name)
    .getOrElse('Guest');

Either

For handling functions that can fail, representing success or error.

    ? Either.Left('Error: Division by zero')
    : Either.Right(a / b);

State Management

FlexNet uses a functional, Redux-inspired approach to state management that emphasizes immutability and pure functions. All state changes are predictable and flow in a single direction.

State Management Principles

  • 1

    Single Source of Truth

    The entire state of your application is stored in a single object tree within a single store.

  • 2

    State is Read-Only

    The only way to change the state is to dispatch an `action`, an object describing what happened.

  • 3

    Changes are made with Pure Functions

    To specify how the state tree is transformed by actions, you write pure `reducers`.

Here's a basic example of the state management system:

import { createStore } from '../../systems/state/functions.js';

// 1. The reducer is a pure function: (state, action) => newState
const counterReducer = (state = 0, action) => {
    switch (action.type) {
        case 'INCREMENT':
            return state + 1;
        case 'DECREMENT':
            return state - 1;
        default:
            return state;
    }
};

// 2. Create a store with the reducer and initial state
const store = createStore(counterReducer, 0);

// 3. Subscribe to changes to update the UI
store.subscribe(() => {
    console.log('New state:', store.getState());
    // In a real app, you would re-render your components here
});

// 4. Dispatch actions to trigger state changes
store.dispatch({ type: 'INCREMENT' }); // Logs: New state: 1
store.dispatch({ type: 'INCREMENT' }); // Logs: New state: 2
store.dispatch({ type: 'DECREMENT' }); // Logs: New state: 1

Effect System

The Effect System provides a way to handle side effects (like API calls, timers, or DOM manipulation) in a controlled, functional, and testable manner. Effects are contained values that describe a side effect but don't execute it until explicitly told to.

Effect System Principles

  • Isolation: Side effects are isolated from your pure business logic.
  • Purity: Creating an effect is a pure operation. The effect is just a value until it's run.
  • Composition: Effects can be composed and transformed using `map` and `chain`.

Example of creating and running an effect:

import { Effect } from '../../systems/effects/functions.js';

// Create an effect that describes fetching a user
const fetchUser = id => Effect.of(() => fetch(`https://api.example.com/users/${id}`));

// `map` transforms the eventual result of the effect
const getJson = response => Effect.of(() => response.json());

const getUserAsJson = id =>
    fetchUser(id)
        .chain(getJson) // chain is used for effects that return effects
        .map(user => user.name); // map is used for pure transformations

// Run the effect to execute it
const effect = getUserAsJson(1);
const userName = Effect.run(effect); // This triggers the fetch call

userName.then(name => console.log(name));

Putting It All Together

Let's see how these concepts work together in a simple counter application. This demonstrates the full, unidirectional data flow of a FlexNet JSX application.

import { createElement, render } from '../../systems/render/functions.js';
import { createStore } from '../../systems/state/functions.js';
import { Effect } from '../../systems/effects/functions.js';

// 1. STATE: Define a reducer and create a store
const counterReducer = (state = 0, action) => {
  switch (action.type) {
    case 'INCREMENT': return state + 1;
    case 'DECREMENT': return state - 1;
    default: return state;
  }
};
const store = createStore(counterReducer, 0);

// 2. EFFECTS: Create a reusable effect
const logEffect = message => Effect.of(() => {
  console.log(`[EFFECT] ${message}`);
});

// 3. VIEW: Create a component as a pure function
const Counter = ({ count }) => {
  // Log every render as a side effect
  Effect.run(logEffect(`Rendering counter with count: ${count}`));
  
  return createElement('div', null,
    createElement('h1', null, `Count: ${count}`),
      createElement('button', {
      onClick: () => store.dispatch({ type: 'INCREMENT' })
    }, '+'),
      createElement('button', {
      onClick: () => store.dispatch({ type: 'DECREMENT' })
    }, '-')
  );
};

// 4. RENDER: Create a function to render the UI based on state
const renderApp = () => {
  const state = store.getState();
    render(
    createElement(Counter, { count: state }),
      document.getElementById('root')
    );
};

// 5. SUBSCRIBE: Subscribe the renderer to store updates
store.subscribe(renderApp);

// Initial render
renderApp();
