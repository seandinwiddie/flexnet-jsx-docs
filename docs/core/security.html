Security

FlexNet JSX is designed with security as a core principle. Its functional architecture provides built-in protections against common web vulnerabilities by promoting explicit data handling, immutability, and pure functions.

Core Security Principles

  • Explicit Sanitization: Content is not rendered as HTML by default. The framework provides explicit functions to sanitize input, preventing Cross-Site Scripting (XSS).
  • Functional Validation: All input validation is handled through pure functions that return explicit types like Either, making error handling robust and preventing invalid data from propagating through the system.
  • Immutable State: State cannot be mutated directly, which prevents a large class of vulnerabilities related to unauthorized or accidental state modification.
  • No Direct DOM Access: By managing rendering through a virtual DOM, the framework limits direct DOM manipulation, which is a common vector for security issues.

XSS Prevention

Cross-Site Scripting (XSS) is prevented by treating all strings as text, not HTML, by default. To render content safely, you must use the provided sanitize function.

Explicit Content Sanitization

import { sanitize } from './src/security/functions.js';
import { createElement } from './src/systems/render/functions.js';

const userContent = '<img src=x onerror=alert(1)>';

// Sanitize the input before rendering
const sanitizedContent = sanitize(userContent);

// This will render the string as text, not as an HTML element
const element = createElement('div', null, sanitizedContent);

// The resulting HTML will be:
// <div>&lt;img src=x onerror=alert(1)&gt;</div>

Input Validation

FlexNet provides two primary pure functions for validating data, both of which return an Either type to explicitly handle success and failure cases.

Predicate-Based Validation with `validateInput`

Use validateInput for simple checks on a single piece of data.

import { validateInput } from './src/security/functions.js';

const isNotEmpty = s => s.length > 0;
const validateUsername = validateInput(isNotEmpty);

const result1 = validateUsername('username'); // -> Either.Right('username')
const result2 = validateUsername('');      // -> Either.Left('Invalid input')

Schema-Based Validation with `validate`

Use validate from the validation feature for complex objects.

import { validate } from './src/features/validation/functions.js';
import { Result } from './src/core/types/result.js'; // Assuming validator returns a Result

// Dummy validator functions that return a Result
const isLongEnough = (str) =>
  str.length >= 8
    ? Result.Ok(str)
    : Result.Error('Password must be at least 8 characters');

// Define validation schema
const userSchema = {
  password: isLongEnough,
};

const validateUser = validate(userSchema);
const validationResult = validateUser({ password: 'short' });

// validationResult would contain the error from the validator

Secure State Management

FlexNet's immutable, reducer-based state management provides security benefits by preventing direct mutation of state and ensuring all changes are explicit and traceable.

Example: Secure State Updates

The correct pattern is to validate input *before* dispatching an action to the store. This ensures only valid data can trigger a state change.

import { createStore } from './src/systems/state/functions.js';
import { validateInput } from './src/security/functions.js';
import { Either } from './src/core/types/either.js';

// Reducer and initial state
const initialState = { user: null };
const authReducer = state => action => {
    if (action.type === 'LOGIN_SUCCESS') {
        return { ...state, user: action.payload };
    }
    return state;
};
const store = createStore(authReducer, initialState);

// Validation predicate
const isValidUsername = name => typeof name === 'string' && name.length > 3;

// Secure login handler
function handleLogin(username) {
  const validationResult = validateInput(isValidUsername)(username);
  
  // Only dispatch an action if validation succeeds
  validationResult.fold(
    (error) => console.error('Validation failed:', error),
    (validUsername) => {
        console.log('Validation succeeded, dispatching action...');
        store.dispatch({ type: 'LOGIN_SUCCESS', payload: { name: validUsername } });
    }
  );
}

handleLogin('Alice'); // Dispatches action
handleLogin('No');    // Logs validation error

Content Security Policy (CSP)

While FlexNet's rendering system prevents DOM-based XSS, it is still crucial to implement a Content Security Policy (CSP) as a defense-in-depth measure. A strong CSP can prevent a wide range of injection attacks by specifying which sources of content are trusted.

An example of a restrictive CSP header is:

Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' https://cdn.tailwindcss.com; img-src 'self';

Security Best Practices

  • Always validate and sanitize data at the boundaries of your application.
  • Use HTTPS for all connections.
  • Follow the principle of least privilege for API access and data handling.
  • Keep dependencies updated to prevent known vulnerabilities.
  • Implement proper logging and monitoring for security events.
