Security

FlexNet JSX is designed with security as a core principle. Its functional architecture, as detailed in the Architecture Overview, provides built-in protections against common web vulnerabilities by promoting explicit data handling, immutability, and pure functions.

Core Security Principles

  • Explicit Sanitization: Content is not rendered as HTML by default. The framework provides explicit functions to sanitize input, preventing Cross-Site Scripting (XSS), as defined in the API Reference.
  • Functional Validation: All input validation is handled through pure functions that return explicit types like Either, making error handling robust and preventing invalid data from propagating through the system. This pattern is explained in the Error Handling guide.
  • Immutable State: State cannot be mutated directly, which prevents a large class of vulnerabilities related to unauthorized or accidental state modification. See the State Management guide for more.
  • No Direct DOM Access: By managing rendering through a virtual DOM, the framework limits direct DOM manipulation, which is a common vector for security issues.

XSS Prevention

Cross-Site Scripting (XSS) is prevented by treating all strings as text, not HTML, by default. The createElement function automatically handles escaping. For an additional layer of security, the framework provides a sanitize function that strips potentially harmful characters from a string entirely.

import { sanitize } from '../../security/functions.js';
import { createElement } from '../../systems/render/functions.js';

const userContent = '';

// The sanitize function returns an Either, so we handle both cases.
// It removes '<' and '>' characters.
const sanitizedContent = sanitize(userContent).getOrElse('');

// sanitizedContent is now "img src=x onerror=alert(1)"
const element = createElement('div', null, sanitizedContent);

// The resulting innerHTML will be:
// <div>img src=x onerror=alert(1)</div>
// The browser will not interpret the string as an HTML element.

Input Validation

FlexNet provides two primary pure functions for validating data, both of which return an Either type to explicitly handle success and failure cases. These are defined in the Security Functions API.

Predicate-Based Validation with `validateInput`

Use validateInput for simple checks on a single piece of data.

import { validateInput } from '../../security/functions.js';

const isNotEmpty = s => s.length > 0;
const validateUsername = validateInput(isNotEmpty);

const result1 = validateUsername('username'); // -> Either.Right('username')
const result2 = validateUsername('');      // -> Either.Left('Invalid input')

Schema-Based Validation with `validate`

Use validate from the validation feature for complex objects.

import { validate } from '../../features/validation/functions.js';
import { Result } from '../../core/types/result.js';

// Dummy validator functions that return a Result
const isLongEnough = (str) =>
  str.length >= 8
    ? Result.Ok(str)
    : Result.Error('Password must be at least 8 characters');

// Define validation schema
const userSchema = {
  password: isLongEnough,
};

const validateUser = validate(userSchema);
const validationResult = validateUser({ password: 'short' });

// validationResult would contain the error from the validator

Secure State Management

FlexNet's immutable, reducer-based state management provides security benefits by preventing direct mutation of state and ensuring all changes are explicit and traceable, as detailed in the State Management guide.

Example: Secure State Updates

The correct pattern is to validate input *before* dispatching an action to the store. This ensures only valid data can trigger a state change.

import { createStore } from '../../systems/state/functions.js';
import { validateInput } from '../../security/functions.js';
import { Either } from '../../core/types/either.js';

// Reducer and initial state
const initialState = { user: null };
const authReducer = state => action => {
    if (action.type === 'LOGIN_SUCCESS') {
        return { ...state, user: action.payload };
    }
    return state;
};
const store = createStore(authReducer, initialState);

// Validation predicate
const isValidUsername = name => typeof name === 'string' && name.length > 3;

// Secure login handler
function handleLogin(username) {
  const validationResult = validateInput(isValidUsername)(username);
  
  // Only dispatch an action if validation succeeds
  validationResult.fold(
    (error) => console.error('Validation failed:', error),
    (validUsername) => {
        console.log('Validation succeeded, dispatching action...');
        store.dispatch({ type: 'LOGIN_SUCCESS', payload: { name: validUsername } });
    }
  );
}

handleLogin('Alice'); // Dispatches action
handleLogin('No');    // Logs validation error

Content Security Policy (CSP)

While FlexNet's rendering system prevents DOM-based XSS, it is still crucial to implement a Content Security Policy (CSP) as a defense-in-depth measure. A strong CSP can prevent a wide range of injection attacks by specifying which sources of content are trusted. For more information, refer to the main Security Practices document.

An example of a restrictive CSP header is:

Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' https://cdn.tailwindcss.com; img-src 'self';

Security Best Practices

  • Always validate and sanitize data at the boundaries of your application.
  • Use HTTPS for all connections.
  • Follow the principle of least privilege for API access and data handling.
  • Keep dependencies updated to prevent known vulnerabilities.
  • Implement proper logging and monitoring for security events.
